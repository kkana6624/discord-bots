# システム設計書: Discord AND条件ロール自動管理Bot

## 1. プロダクト概要

### 1.1 目的と解決する課題

Discordの標準機能（OR条件による権限評価）の制約を解消し、「特定のロール群（ロールAかつロールB）が揃ったユーザーに対してのみ、アクセス権限となる第3のロール（ロールC）を自動付与・剥奪する」ことに特化したバックグラウンドマイクロサービスです。

### 1.2 コア機能

* **リアルタイム自動付与/剥奪**: メンバーのロール変更をWebSocket経由で検知し、条件に合致した瞬間に即時で状態を反映します。
* **フェイルオープン防止（自動修復）**: ネットワーク瞬断やAPIレートリミットによる「ロール権限の消し忘れ」をシステム的に排除するため、定期的な全メンバーの監査（Reconciliation）をバックグラウンドで実行します。

## 2. アプリケーション設計 (Elixir / OTPアーキテクチャ)

本システムはElixirとOTPの特性を活かし、プロセス内部でクラッシュ許容と自動復旧を完結させる「Let it crash」モデルを採用します。

### 2.1 スーパービジョンツリー（プロセス監視構造）

イベント駆動による即時反映と、バッチ駆動による確実な状態調停を、独立したActorとして並行稼働させます。

```text
[Application] (最上位モジュール)
 └── [Root Supervisor] (戦略: one_for_one)
      ├── [Nostrum Application] (接続管理・Gateway APIイベント受信)
      ├── [Event Consumer] (GenStageベース: ロール変更イベントの即時評価)
      └── [Reconciliation Worker] (GenServer: 定期全メンバー監査と不整合の強制修復)

```

### 2.2 権限評価ロジック（純粋関数）

ローカルで期待される状態を算出し、差分がある場合のみAPIリクエストを発行して負荷を最小化します。

* **付与 (Add)**: ロールAとBを所持、かつロールCを未所持の場合。
* **剥奪 (Remove)**: ロールAまたはBが欠如しており、かつロールCを所持している場合。

## 3. インフラ・セキュリティ設計 (OCI基盤)

可用性とセキュリティを両立しつつ、運用保守のコストを極小化するため、「単一コンテナ運用」**と**「インフラ層でのシークレット動的注入」を採用します。イベントの重複受信によるAPI Rate Limitペナルティを防ぐため、複数コンテナ（Active-Standby構成等）は採用しません。

### 3.1 実行環境

* **クラウドプロバイダ**: Oracle Cloud Infrastructure (OCI) Always Free枠
* **コンピュート**: ARMベース (Ampere A1) または AMD Microインスタンス
* **コンテナランタイム**: Podman (またはDocker)
* **可用性担保**: コンテナマネージャー側で `--restart=always` を設定し、ホスト再起動時やVMクラッシュ時の自動復帰を担保します。

### 3.2 シークレット管理と動的注入 (OCI Vault連携)

環境変数ファイル（`.env`）などに機密情報を平文保存することを避け、OCI Vault（Key Management）を利用してエンタープライズ水準のクレデンシャル管理を行います。

* **Instance Principalの利用**:
対象インスタンスを「動的グループ」に所属させ、OCI VaultのSecret Bundleを読み取るIAMポリシーを付与します。これにより、インスタンス内にアクセスキーを配置する必要がなくなります。
* **Entrypoint層での注入（The Twelve-Factor App準拠）**:
Elixirアプリケーション自体は特定のクラウド機能に依存させず、「環境変数からトークンを読み込むだけ」のポータビリティの高い状態を保ちます。
コンテナの `entrypoint.sh` 起動時に `oci-cli` を利用してVaultからシークレットを取得し、メモリ上で環境変数として展開してからアプリケーション（BEAM）を起動します。

**【Entrypointの概念イメージ】**

```bash
#!/bin/sh
# OCI CLIを利用し、Instance Principal認証でVaultからシークレットを取得
SECRET_BUNDLE=$(oci secrets secret-bundle get --secret-id ${VAULT_SECRET_OCID} --auth instance_principal)

# ペイロードからBase64文字列を抽出し、デコードして環境変数としてメモリ空間にエクスポート
export DISCORD_TOKEN=$(echo $SECRET_BUNDLE | jq -r '.data."secret-bundle-content".content' | base64 -d)

# アプリケーションプロセスへの置換（環境変数を引き継ぐ）
exec mix run --no-halt

```

## 4. 異常系ハンドリングと回復戦略

単一コンテナ構成の弱点を、アプリケーションレベルの自己修復能力で補完します。

1. **Gateway(WebSocket)の一時的な切断**:
* ライブラリ(Nostrum)がタイムアウトを検知しプロセスをクラッシュ。Supervisorが即座に再接続を試みます。


2. **ダウンタイム中のイベント取りこぼし**:
* アプリケーション起動直後、および定期的に動作する「Reconciliation Worker」が最新の状態をフェッチして不整合を強制修復（自己修復）するため、手動での介入は不要です。


3. **API Rate Limit (HTTP 429) への抵触**:
* 大量のユーザーのロールが同時に変更された場合、ElixirのHTTPクライアント層で `Retry-After` ヘッダを解釈し、APIへのリクエストを適切にバックオフ（スリープ）してペナルティを回避します。
